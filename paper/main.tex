\begin{abstract}Coding agents are increasingly evaluated on realistic software-engineering tasks, but many high-value domains require code to work inside running frameworks with state, events, assets, and interacting systems. Game-engine programming is a demanding instance of this problem. We present \benchmarkname, a benchmark for evaluating coding agents on scoped C++ edits inside Unreal Engine 5 projects, built from nine real game repositories. The evaluated set contains 120 tasks covering gameplay mechanics, multiplayer behavior, AI and world orchestration, animation and movement code, UI and session code, loading behavior, online-service integration, persistence, data serialization, XR behavior, and rendering-oriented plugin code. Each task requires the model to edit native C++ files, compile, and satisfy behavioral tests inside a functioning Unreal project. Across ten evaluated configurations, pass@1 remains below 30\%. The failures are not primarily superficial syntax errors: many runs compile and recover partial behavior, but still fail because of authority mistakes, lifecycle bugs, state-synchronization errors, game-state mistakes, or incomplete integration with surrounding systems. \benchmarkname shows that current coding agents still struggle with runtime-integrated C++ changes, making game-engine benchmarks a valuable complement to general software-engineering evaluations.
\end{abstract}

\section{Introduction}
Coding agents are increasingly evaluated on realistic software-engineering tasks, but many benchmarks still undermeasure domains where correctness depends on a running framework rather than on local code behavior alone. Game-engine programming is one such domain. A generated C++ patch may compile and look plausible while still failing because it runs on the wrong machine, updates only local state, misses an object-lifecycle transition, or breaks an interaction with another engine system.

This matters beyond entertainment software. Game engines provide real-time simulation, rendering, physics, interaction, networking, and asset pipelines, making them useful in simulation, digital twins, XR, robotics prototyping, virtual production, and interactive training. Game development is a practical source of benchmark tasks for this broader class of software because game repositories are mature, executable, stateful, and often publicly available.

The difficulty is that game-engine tasks require models to satisfy both ordinary C++ requirements and framework-level behavioral contracts. In a networked game, a small change may need to run on the authoritative server, synchronize state to clients, update UI only for the local player, and clean up correctly when actors are destroyed or reused. These requirements are easy to describe at a high level but hard to verify completely: a test may check one interaction path while a valid implementation must preserve behavior across several classes, callbacks, and runtime states.

Existing coding benchmarks capture important parts of agent capability, including isolated function synthesis, issue resolution in software repositories, long-horizon codebase tasks, and multimodal game construction. They do not directly evaluate scoped native C++ edits inside functioning game-engine projects, where correctness depends on engine execution, networking behavior, object lifecycles, subsystem initialization, and integration with existing gameplay code. \benchmarkname addresses this gap using Unreal Engine projects as an executable testbed.

\benchmarkname contains 120 tasks drawn from nine real Unreal Engine repositories. Each task gives the model a buildable start state, scoped editable C++ files, and a behavior specification. After the model finishes, tests are injected and run through Unreal's Play-in-Editor automation, and judge auditing determines whether the implementation satisfies the requested behavior rather than merely matching a reference solution. Across ten evaluated configurations, pass@1 remains below 30\%. Many failed runs still recover substantial partial behavior, but the failures cluster around recurring runtime and integration patterns, including authority mistakes, state-synchronization errors, object-lifecycle bugs, initialization errors, and incomplete integration with surrounding game systems.

We make three contributions. First, we introduce a 120-task benchmark for evaluating C++ changes that must integrate into functioning Unreal Engine projects. The models only edit native source files.\footnote{Some source projects use non-code assets during execution, but benchmarked edits are restricted to native C++ files.} Second, we define an evaluation protocol that combines scoped file edits, runtime tests, and judge auditing to score behavioral correctness rather than reference similarity. Third, we analyze why current agents fail, showing that low pass@1 reflects structured runtime and integration failures rather than superficial syntax errors alone. By grounding these failures in executable game projects, \benchmarkname provides a concrete measure of current coding-agent capability on game-engine programming tasks and highlights where agents need stronger evaluation and training signal.

\section{Related Work}

\textbf{Software Engineering Benchmarks}. Software engineering has become a central domain for evaluating LLM-based coding agents. Earlier code-generation benchmarks such as HumanEval and MBPP measure isolated function synthesis \citep{chen2021evaluating,austin2021programsynthesis}. SWE-bench evaluates whether models can resolve real GitHub issues in conventional software repositories \citep{jimenez2023swebench}; SWE-bench-Live and SWE-rebench emphasize freshness, continuous task collection, and contamination-resistant evaluation \citep{zhang2025swebenchlive,badertdinov2025swerebench}; Terminal-Bench evaluates agents on realistic command-line tasks that require operating inside an execution environment \citep{merrill2026terminalbench}. These benchmarks establish bug fixing and environment-level execution as important alternatives to isolated coding problems. \benchmarkname targets a different part of the software-engineering spectrum by evaluating larger implementation tasks inside functioning runtime systems, using game projects as the concrete execution environment. Across the 120 evaluated tasks, reference solutions add a mean of 504 lines and a median of 363 lines within the editable source files, making the tasks larger than the SWE-bench Verified and SWE-bench Pro reference edits reported by DeepSWE \citep{huang2026deepswe}. This shifts the evaluation from localized bug repair toward implementing missing behavior inside an existing runtime system.

\textbf{Frontier Agent Benchmarks}. Recent agent benchmarks have shifted from overspecified implementation specifications to realistic behavior-focused instructions that require deeper reasoning and understanding. DeepSWE evaluates frontier coding agents on original, long-horizon software engineering tasks with broad repository coverage and behavior-focused verifiers \citep{huang2026deepswe}. ProgramBench evaluates whether agents can rebuild complete software projects from documentation and executable behavior, showing that agents often produce codebases that behave partially correctly but diverge sharply from human-written implementations \citep{yang2026programbench}. FrontierCode shares the emphasis on realistic codebase tasks, but uses maintainer-authored tasks and rubric-based grading to shift the target from behavioral correctness alone toward mergeability, evaluating whether generated code would meet production codebase standards \citep{lu2026frontiercode}. \benchmarkname follows this direction while focusing on runtime-integrated C++ programming: native edits inside functioning Unreal Engine projects, where correctness depends on framework behavior such as authority, state synchronization, object lifecycle, and subsystem initialization.

\textbf{Game Development Benchmarks}. GameDevBench establishes game development as a meaningful domain for agent evaluation with tasks derived from web and video tutorials, emphasizing multimodal game-development work such as visual scenes, shaders, sprites, and animations \citep{chi2026gamedevbench}. AutoUE studies automated 3D game generation in Unreal Engine through multi-agent systems and automated play-testing, but targets end-to-end game creation rather than constrained C++ changes inside existing projects \citep{yin2026autoue}. GameDevBench does not target multiplayer or server/client correctness, which are central sources of failure in networked game development. \benchmarkname instead evaluates native C++ changes inside existing Unreal projects. This focuses the benchmark on runtime-integrated programming, where code must work with existing gameplay logic, networking, and engine architecture.

\textbf{Game-Engine Runtime Concepts}. We use the following terms throughout the paper. \emph{Multiplayer authority} refers to the rule that authoritative gameplay decisions should run on the server rather than on an individual client; this follows the client/server model used by Unreal networking \citep{epic2026networking}. \emph{Replication} is Unreal's mechanism for synchronizing gameplay state and procedure calls between server and clients; property replication sends server-side updates to connected clients, which then apply those values to their local actor instances \citep{epic2026networking,epic2026replicateactorproperties}. \emph{Object lifecycle} refers to the sequence of actor and object events around spawning, initialization, gameplay start, teardown, destruction, and garbage collection \citep{epic2026actorlifecycle}. \emph{Subsystem architecture} refers to Unreal-managed systems such as game-instance, world, and local-player subsystems that the engine creates, initializes, and exposes to gameplay code \citep{epic2026subsystems}. These behaviors must work together to produce a correct solution to a development task. This emphasis on cross-system behavior is consistent with prior work showing that game-engine subsystems are tightly coupled and hard to understand in isolation \citep{ullmann2023visualising}. A solution that compiles can still fail if it runs on the wrong machine, updates only local state, cleans up too late, or registers a component after another system expects it to exist.

\section{Benchmark Design}
Every task gives the model a buildable start state, a list of native C++ files available for modification, and a public set of text instructions that define observable task-specific behavior. Tests are kept out of the solver workspace and added only after the model finishes, which reduces leakage from the evaluation logic into the solve phase.

Before including a task in the benchmark, we use a multi-step authoring process with two distinct checks. \emph{Test validation} checks that the task package loads, the start project builds, the reference solution passes the tests, and the start state fails those tests on the intended missing behaviors. \emph{LLM-as-a-Judge review} runs a strong coding model on the task and asks an LLM judge to review the saved result against the public behavior specification, test outcomes, model edits, and reference solution. This review identifies tests that are under-specified, too narrow, or over-constrained relative to the requested behavior. When the review reveals a problem, we revise the task or test suite and rerun validation. We use \emph{task calibration} to refer to this overall pre-release process of validating the tests, reviewing strong-model outputs, revising the task, and rerunning validation. Figure~\ref{fig:task-authoring} summarizes the process.

\section{Sample Tasks}
Two examples illustrate the kinds of Unreal work included in the current evaluated task set. We choose these examples because both remain pass@1 failures for all evaluated configurations and expose different forms of engine-level coordination.

Task \texttt{1} (Zombie System) is a broad multiplayer gameplay task in HordeTemplate. It requires coordinated edits across eight source files so zombies spawn through the round system, update the alive-zombie counter, detect and pursue living players, stop acting after death, enforce melee range, award kill and headshot rewards, and expose attack/death feedback to clients. The task is difficult because correctness depends on several systems agreeing at once: AI control, game-mode round progression, player-state rewards, server-authoritative damage, replicated feedback, and spawn/patrol metadata.

Task \texttt{19} (Generated Map Orchestrator) is narrower in file count but still integration-heavy. It requires a generated map actor to populate a procedural grid at round start, reuse pooled actors across regenerations, send one ready signal after local generation completes, replicate map-size changes, and keep peers synchronized. This task is difficult because the implementation must coordinate generation, actor pooling, replication, editor preview behavior, and readiness signaling without firing too early, firing twice, or leaking actors across regenerations. Together, these examples show that the benchmark's hardest tasks are not defined only by edit size; they require consistent behavior across interacting Unreal systems.

\section{Evaluation Protocol}
The headline metric used in this paper is \emph{pass@1}: the fraction of tasks solved by a single model attempt. A run counts as a pass@1 success when the LLM judge concludes that the model implementation satisfies the requested behavior. This LLM-judged definition is necessary because the tasks often span several engine systems, making it difficult for any test suite to cover every valid implementation and every relevant interaction path. Test results remain the primary execution signal, but they can understate model capability when they miss a valid behavior-preserving solution.

The LLM review follows the LLM-as-a-Judge pattern: an LLM receives the behavioral specification, test source, concrete test outcome and assertion errors, model-edited files, and reference solution, then decides whether the implementation satisfies the public task \citep{zheng2023judging}. Its role is not to check whether the model copied the reference solution, but to decide whether the model implementation satisfies the requested behavior and whether the test result correctly reflects that fact. To reduce self-judging effects, the protocol is designed to support cross-family judging when multiple model families are available.

Each run is executed in an isolated temporary workspace copied from the task's public \texttt{start/} state. The model edits only the task's allowed native C++ files in that workspace. After the solve phase, tests are injected and executed through Unreal's automation framework in Play-in-Editor (PIE) listen-server mode, and the benchmark saves the resulting workspace, logs, and judge output for auditability. This separation reduces leakage, preserves reproducibility, and keeps the public task, tests, and judge review distinct.

\subsection{Full-Task Success Remains Low, But Partial Progress Is Common}
Figure~\ref{fig:pass-rate-lenses} shows the headline pass rate together with complementary scoring lenses. The task-level pass rate stays low, but this does not mean that model outputs are usually unrelated to the requested behavior. The strongest configurations satisfy a much larger fraction of individual requirements than they solve as complete tasks: \texttt{gpt-5.5} at \texttt{xhigh} reaches 28.7\% pass rate but 78.7\% mean requirement satisfaction, while \texttt{claude-opus-4-8} at \texttt{max} reaches 23.5\% pass rate and 78.6\% mean requirement satisfaction. This gap is important for interpreting the benchmark. Many failed runs contain plausible C++ and recover substantial local behavior, but they miss one or more interactions needed for the full Unreal task to work.

The scoring-lens split also explains why a single pass-rate number is incomplete for this benchmark. A model can pass many hidden checks while still failing the task because one runtime dependency is wrong: a client lacks the replicated state needed by a UI update, an object is not cleaned up across reuse, or a subsystem registers after another class expects it to exist. These are not superficial syntax failures. They are failures to connect correct-looking local code to the surrounding game system.

\subsection{Compilation Is Not The Main Bottleneck For The Strongest Wrappers}
Figure~\ref{fig:failure-compile} separates passes, test failures, judge-disputed cases, and compile failures. For the GPT and Claude configurations, compile rates are high: the GPT configurations compile 95.8\% of runs, \texttt{claude-opus-4-8} at \texttt{max} compiles 95.8\%, and the Claude high-effort configurations compile between 94.2\% and 95.8\%. Kimi also compiles most runs at 90.8\%. For these configurations, most failures happen after code has compiled and tests have run. In other words, the hard part is usually not producing syntactically valid Unreal C++; it is producing code that behaves correctly inside the existing runtime system.

\subsection{Task Difficulty Is Structured Across Models}
The unsolved tasks are not random. Across the 120-task set, 46 tasks are failed by every evaluated configuration, while 74 are solved by at least one configuration. Only one task is solved by all ten configurations. This distribution matters because it separates two kinds of evidence. Tasks solved by at least one model show that the harness and specification are often sufficient for successful solutions. Tasks failed by every model identify shared capability gaps in current agents.

\subsection{Concrete Failure Patterns}
The most informative failures are the ones where models produce reasonable-looking Unreal code but miss the runtime contract. Task \texttt{1} (Zombie System) remains unsolved by every evaluated configuration. The task requires AI control, round-state updates, server-authoritative damage, player-state rewards, replicated feedback, and spawn metadata to agree. A partial implementation can add zombie spawning or damage logic and still fail because the alive counter, patrol fallback, kill rewards, or client-visible feedback does not line up with the rest of the round system.

Task \texttt{19} (Generated Map Orchestrator) is another universal failure. The implementation must generate a grid, reuse pooled actors, replicate map-size changes, and emit exactly one readiness signal after local generation completes. Failures here are coordination failures: callbacks fire too early or not at all, pooled actors leak state across regenerations, or readiness tokens become inconsistent across repeated map builds. The code can contain the expected functions and still fail because the ordering and lifecycle behavior are wrong.

Persistence tasks expose a related weakness. The SPUD tasks \texttt{76--80} are failed by every configuration in the current results. These tasks require runtime actors, destroyed actors, serialized properties, streaming levels, and save/load lifecycle code to remain consistent across sessions. This is a different kind of difficulty from writing an isolated algorithm. The model must preserve identity and state across engine-managed load and teardown boundaries, where a locally plausible edit can silently break restoration behavior.

The same pattern appears in narrower model-specific failures. In active-item replication tasks, models sometimes add an \texttt{OnRep} hook but leave the client without the replicated source state needed to update visible UI. In player-controller and menu tasks, models can omit local-controller checks, allowing client-specific browsing or playback state to run on the wrong controller instance. In ability and action-system tasks, models often recover most declarations but miss constructor defaults, activation timing, or teardown behavior. These examples explain why the requirement-satisfaction rates in Figure~\ref{fig:pass-rate-lenses} are much higher than full-task pass rates: the models often find the right local ingredients, but fail to connect them into a correct runtime behavior.

\section{Limitations and Next Steps}
The current task set is a starting point rather than complete coverage of Unreal C++ development. Its strongest coverage is gameplay-facing C++ systems inside existing projects: combat, inventory, networking, actor lifecycle, player/session flows, subsystem integration, and data/configuration behavior. It is much weaker for other C++ work that also matters in game engineering, including rendering, physics, audio, memory, performance, platform support, editor tools, build systems, and security-sensitive code.

The source repositories also shape what the benchmark can represent. Open-source Unreal projects tend to expose gameplay and framework code more readily than proprietary engine modifications, rendering work, platform integrations, performance infrastructure, or security-sensitive systems. As a result, the current evaluated set should not be interpreted as measuring every kind of professional C++ game engineering.

The execution harness introduces another boundary. Running hidden tests through PIE listen-server automation is a practical way to test runtime behavior, authority, replication, and lifecycle issues, but it does not fully reproduce dedicated server fleets, console hardware, performance profiling, certification requirements, large QA matrices, or long-running live operations. Hidden tests also remain a sample of the desired behavior rather than an exhaustive proof, which is why the paper uses judge calibration and manual audit.

Future expansion should add more tasks where correctness depends on coordination across multiple engine systems, including multiplayer state synchronization, object lifecycle edge cases, subsystem interactions, large combat or ability systems, and plugin-level integration. A separate expansion should target currently thinner C++ areas such as rendering, physics, audio, performance, editor tooling, build/platform code, and security-sensitive gameplay infrastructure. These task types would broaden representativeness beyond the current gameplay-facing systems slice while preserving the engine-integrated runtime failures that current agents still make.

Two next steps will strengthen the current results. First, the evaluated models run through different wrappers, so observed pass rate differences partly mix model capability with wrapper behavior. Running all models through a common wrapper will make the comparison cleaner. Second, LLM review changes some test failures into pass@1 successes, showing that the tests do not always capture the full behavior space of these larger tasks. Continued test validation will reduce dependence on judge correction and make the execution signal clearer.

\section{Conclusion}
\benchmarkname marks a concrete step toward benchmark design for AI-assisted game-engine programming. Instead of evaluating simpler execution outcomes and syntactic accuracy, our benchmark aligns coding outputs with specified behavioral requirements, requiring a greater understanding of the surrounding game system by the agent. The tasks are extracted from functional Unreal Engine repositories, enabling the benchmark to cover production-like C++ runtime behaviors while leaving several kinds of C++ game programming for future expansion. The 120-task set shows that frontier coding agents do not solve this gameplay-facing C++ systems slice by default. The remaining failures are engine-semantic and integration-heavy rather than superficial syntax or compile errors. While our tasks are specific to Unreal Engine, the gaps in model performance reflect broader challenges in engine- and framework-integrated software development, where generated code must work inside an existing project, follow framework rules about where and when code should run, and preserve behavior under realistic runtime conditions. That makes \benchmarkname useful today as a focused measure of difficult Unreal C++ work and a foundation for broader game-development evaluation later.
